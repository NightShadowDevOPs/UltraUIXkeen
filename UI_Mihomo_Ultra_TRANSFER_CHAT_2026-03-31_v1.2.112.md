Стек: Vue 3 + TypeScript + router-agent (shell/cgi на роутере)

Текущие версии:
- UI: v1.2.112
- v1.2.112: второй этап оптимизации роутерной версии — `status` / `qos_status` переведены на более лёгкую пассивную диагностику shaper state; `status` больше не делает 200мс sleep для CPU-замера, а фронт схлопывает короткоживущие запросы к agent
- router-agent: 0.6.25

Что уже сделано по оптимизации:
- `v1.2.110`: из роутерной версии убран отдельный раздел `Mihomo`; старый путь `/mihomo` редиректит на `Router`
- `v1.2.111`: скрытые секции `Router` теперь размонтируются вместо фоновой жизни через `v-show`; polling для router cards / traffic / QoS / Tasks стал реже и останавливается на скрытой вкладке браузера
- `v1.2.112`: базовые router-agent endpoints облегчены — `status` и `qos_status` больше не пытаются чинить shaper state на каждый опрос; короткий cache/coalescing уменьшает параллельную нагрузку от нескольких карточек

Что важно по текущему состоянию:
- роутерный проект теперь фокусируется только на runtime/traffic/QoS/shaping/hosts/diagnostics
- тяжёлый Mihomo config workspace считается замороженной веткой и не участвует в активной роутерной навигации
- shaping path сейчас такой: IFB ingress → `wan-police` через `iptables hashlimit` → legacy `lan-egress`
- на живом роутере IFB на XKeen 1.1.3.9 / kernel 4.9-ndm-5 не поднимается, поэтому `wan-police` fallback остаётся критичным

Что важно не потерять:
- если меняется `router-agent`, синхронизировать версию в `router-agent/install.sh` и в `status` API
- UI на роутере обновляется через сам интерфейс UI, а не через `git pull`
- в командах для роутера добавлять `clear`

Следующий приоритет:
- пройтись по самым тяжёлым agent-командам и отделить лёгкий `status` от расширенной диагностики окончательно
- проверить на живом роутере, что `wan-police` реально ограничивает клиента через активные `shape`-правила и counters
- при необходимости добавить лёгкую встроенную телеметрию agent (частота вызовов и длительность команд), чтобы дальше оптимизировать уже по фактам
